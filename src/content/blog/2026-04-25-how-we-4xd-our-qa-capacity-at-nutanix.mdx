Every QA team I've worked with has the same bottleneck: not enough clusters. Engineers queue up. PRs pile behind pre-check runs. Someone suggests "can we just buy more hardware?" and finance suggests otherwise.

At Nutanix, on the developer productivity team, we hit exactly this wall. But instead of optimizing test execution, we changed what tests ran on. This is how we ended up spinning 4 virtual clusters out of 1 physical cluster, and why it mattered for more than just the hardware budget.

## The wall we hit

Our QA infrastructure looked roughly like this. A fleet of physical clusters, each one dedicated to a specific suite or team. When a developer opened a PR, pre-check runs had to grab a cluster, deploy the build, run tests, tear it down, and hand the cluster back to the queue.

Two things were constantly broken:

**Queue depth.** At peak, a PR could wait hours before a cluster was free. Multiply that across a few hundred engineers and you get a noticeable drag on ship velocity.

**Dirty clusters.** Tests don't always clean up after themselves. Reboots, upgrade paths, failed teardowns, leftover VMs, all of it accumulated. Every few days a cluster would start failing tests for reasons completely unrelated to the PR under test. We'd spend hours debugging "why is this test flaky" before realizing the cluster state was corrupted from three runs ago.

The traditional answer is "buy more hardware." But hardware has a multi-quarter procurement cycle, a rack-and-stack tax, and at our scale, the capex was real. We needed a different lever.

## What we changed

Nutanix's platform was already good at one thing: spinning up virtualized infrastructure fast. So the question became: can we run our tests on virtual clusters deployed on top of physical ones?

The answer, as it turns out, was yes. Each physical cluster could host **3 to 4 virtual clusters** simultaneously. We rebuilt the CI infrastructure around this.

That single shift, physical to virtual, changed the economics in three places at once.

## What it unlocked

**1. A 3-4x multiplier on cluster capacity without buying hardware.**

Every physical cluster was now effectively doing the work of 3 or 4. Across the fleet, that meant roughly 3-4x the effective capacity we had before. Capex flat, throughput up.

**2. Parallel test execution instead of serial.**

The old model was queue-driven: tests waited their turn on a physical cluster. The new model let a PR spin up a fresh virtual cluster on demand, run its tests, and tear down, while three other tests did the same thing on the same physical box. Developer PR runs got dramatically faster because the bottleneck shifted from "wait for cluster" to "run the test itself."

**3. The dirty-cluster problem disappeared.**

Virtual clusters are disposable. Spin one up, run the test, delete it. Every test starts with a pristine cluster. No leftover state, no ghost VMs, no "why is this test flaky on Tuesdays" debugging. That alone saved the team hours every week that used to go into environment forensics.

## The part most people forget: resilience

Virtualization introduced a new risk. If the management plane for those virtual clusters went down, you'd lose visibility into everything running on top. The easy design is "one Prism Central managing all virtual clusters." The right design is distributed.

We attached **multiple Prism Elements to a single Prism Central**, so if any one management node failed, the system continued operating. A broken node becomes a replacement job, not an outage. Business as usual.

This is an easy thing to skip when you're heads-down shipping a migration. It's also exactly the kind of thing that separates a working system from one that stays working at 3am on a Saturday.

## What I took from this project

Three things, all of which I've carried into every TPM role since:

**The highest-leverage QA improvement is often not a QA improvement.** Most teams facing "we don't have enough clusters" reach for parallelization, better test sharding, or smarter queue management. Those are optimizations. The bigger win was changing the substrate the tests ran on. When you can't optimize further inside the current frame, change the frame.

**Disposable beats reusable for test environments.** State is the enemy of reliable tests. Any time you can make an environment cheap to throw away, you eliminate a whole category of flakiness. This generalizes, containerized test runners, ephemeral preview environments, snapshot-based DB resets, all of it is the same principle.

**Distributed beats centralized for anything you depend on.** A single management plane is a single point of failure. Multiple PEs per PC cost more to design but cost nothing to operate. Resilience is cheapest when you build it in, not when you bolt it on.
